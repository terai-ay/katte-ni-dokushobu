## システム概要

### サービス概要

**「勝手に読書部」** は、書籍に特化したコミュニティマーケットプレイスおよび推薦エンジンです。複数の虚構の書店（ストア）を運営する棚として機能し、ユーザーは以下を実行できます：

- **本の出品・購入**: 新しい本、中古本の売買
- **気分ベースの推薦**: AIアシスタント「ぽよ」による本の推薦
- **コミュニティ投票**: リアクション機能で本に投票
- **読みたいリスト**: 読みたい本をコミュニティで記録・共有
- **情報共有**: 新着本のお知らせ掲示板

### コアコンセプト

- **手作り感**: 読書好きが自由に集う場所
- **インクルーシブ**: 認証なし、パスワード不要で参加可能
- **非営利**: 本そのものの流動化を助ける
- **楽しさ重視**: 本探しを冒険に、推薦を助言に

## 主な機能

### 1. マルチストア書店

6つの虚構の書店があり、各ジャンル特性を持ちます：

| ストア | 説明 | アイコン | 特徴 |
|--------|------|--------|------|
| **ブックセンター** | 総合書店 | ? | ジャンル問わずなんでも |
| **インスパイアブックス** | 専門書・ビジネス | ? | 知識・学習志向 |
| **コミメイト** | マンガ専門店 | ? | 漫画・グラフィック |
| **文学のともしび** | 純文学・古典 | ?? | 深い思想・詩集 |
| **星空書房 火星店** | SF・宇宙・未来 | ? | 想像力をかき立てる |
| **闇夜の推理堂** | ミステリー | ? | 謎・冒険・スリル |

各ストアは独立したデータを管理し、ユーザーは好きなストアで本を出品できます。

### 2. ぽよ呼び出し機能

#### ぽよとは

「ぽよ」はシステムのAIアシスタント。画面左下の呼び出しボタン（?）から召喚でき、ユーザーの読書体験を支援します。

#### サブ機能

**① ランダムメッセージ表示**
- ぽよが常に9種類のメッセージをランダムに（6秒ごと）表示
- 例：「今日はどんな本を読もうかな～」「気分を教えてくれたらおすすめするよ！」
- 目的：ユーザーエンゲージメント向上、居心地の良さ

**② 気分ベース本推薦（ランダム表示）**
- 6つの気分カテゴリから選択：
  - **ぼんやり** → ブックセンター、文学のともしび の本からランダムに1冊
  - **熱い** → インスパイアブックス、星空書房の本からランダムに1冊
  - **しんみり** → 文学のともしびの本からランダムに1冊
  - **笑いたい** → コミメイト、ブックセンターの本からランダムに1冊
  - **学びたい** → インスパイアブックスの本からランダムに1冊
  - **なんでも** → 全ストアの本からランダムに1冊
- 気分を選ぶと、該当ストアから本がランダムに選ばれて表示
- 表示内容：表紙画像、書名、著者、投稿者情報、ストアアイコン

**③ 全店おすすめ一覧表示**
- すべての本をストア別グリッド表示
- スクロール可能な一覧で閲覧
- 本をクリックでチャットモーダルが開き、投稿者と直接やり取り可能

### 3. 古本市（フリマ機能）

ユーザーが本の出品・売買を行うマーケットプレイス。

**出品種別**：
- **あげます** (Free Listing)：無料で譲る
- **売ります** (Sell)：価格を設定して販売
- **貸します** (Lend)：一時的に貸し出す（貸出中状態管理機能付き）

**状態管理**：
- 出品後、「取引完了」で非表示化（ソフトデリート）
- パスワード保護オプション：誤削除防止のため、完了時にパスワード入力を要求可能
- 「貸します」出品は「借ります」ボタン押下で「貸出中」ラベル表示、再度押下で「貸出中」解除

**UI特性**：
- カードグリッド表示、常にすべてのカードを表示
- モバイルでも見出しから全カード確認可能
- 投稿者コメント・取引方法表示

### 4. ランキング機能

本に対する5種類のリアクション（?よんだ、??きになる、?ビビビッ、?love、?泣いた）の数を集計し、リアクション種別ごとにランキング表示。

### 5. みんなの読みたいリスト

ユーザーが読みたい本をテキスト入力で登録。
- 書名、著者名（任意）、投稿者名（任意）
- 削除用パスワード保護（任意）
- リアクション機能で「+1」投票可能

### 6. チャット＆リアクション

各本の詳細モーダル内で、投稿者・購入者間のチャット機能。
チャットメッセージに対して5種類の絵文字リアクションを付与可能（右側ボタンクリック、左下に集計表示）。

## 技術アーキテクチャ

### スタック

```
Frontend                    Hosting           Database
┌──────────────────┐      ┌──────────────┐   ┌──────────────┐
│  HTML5/CSS3/JS   │      │ GitHub Pages │   │   Supabase   │
│  (Vanilla)       │  ──→ │  (Static)    │──→│ (PostgreSQL) │
└──────────────────┘      └──────────────┘   └──────────────┘
     (index.html)         (Public Repo)      (REST API)
```

### コンポーネント詳細

**Frontend**
- **言語**: HTML5 + CSS3 + Vanilla JavaScript（フレームワークなし）
- **特徴**:
  - 軽量、依存なし
  - 完全にクライアント側で動作
  - Supabase REST APIを直接呼び出し
  - localStorage を使用した非永続データの一時保存

**Hosting**
- **GitHub Pages**: 無料ホスティング
- **制約**: 静的サイトのみ、Public Repository 必須
- **利点**: CD/CI 自動統合、バージョン管理

**Database**
- **Supabase**: オープンソース Firebase 代替
- **ベース**: PostgreSQL
- **アクセス**: REST API via HTTPS
- **認証**: なし（セキュリティ課題を参照）

### 画像保存方式

- **現在**: Base64 インライン化
- **影響**: Base64 化により文字列データベース容量が増加（元の約1.33倍）
- **課題**: 大規模運用時の容量圧迫

### API通信

すべての CRUD 操作は Supabase REST API 経由：

```javascript
// 例：本を取得
await fetch('https://[project].supabase.co/rest/v1/books?select=*', {
  headers: {
    'apikey': PUBLIC_API_KEY,
    'Authorization': `Bearer ${SESSION_TOKEN}`
  }
})
```

**注意**: 公開リポジトリのため、API キーは JavaScript 内に露出。

## セキュリティ考慮事項

### ?? 識別されたリスク

#### 1. **XSS (クロスサイト スクリプティング)**
- **問題**: ユーザー入力（コメント、書名）が、サニタイズなしでDOMに挿入される可能性
- **影響**: 悪意あるスクリプト埋め込みで、他ユーザーのデータ盗取・改ざん
- **現状**: 低リスク（小規模コミュニティ）
- **対策**: `textContent` 使用、 `innerHTML` 削減、入力バリデーション

#### 2. **API キー露出**
- **問題**: Supabase 公開 API キーが index.html にハードコード
- **影響**: Public Role 権限内で自由な読み書き可能、レート制限なしで DDoS 可能
- **制約**: 公開リポジトリのため対策困難
- **対策**: 
  - バックエンド作成（非公開 API キーを隠蔽）
  - または Supabase 権限管理（RLS）の厳格化

#### 3. **認証・認可なし**
- **問題**: ユーザー識別がなく、誰でも何でも変更可能
- **影響**: 出品取消し、チャット削除、ランキング改ざん
- **対策**: 軽量認証（例：セッショントークン）導入

#### 4. **平文パスワード保存**
- **問題**: 完了・削除時のパスワードが Base64 エンコードのみで保存
- **影響**: データベース漏洩時、全パスワード露出
- **対策**: bcrypt など、1方向ハッシュ関数で保存

#### 5. **RLS (Row Level Security) 未設定**
- **問題**: Supabase テーブルの RLS ポリシーが標準のまま（無制限）
- **影響**: 誰でも他人の出品を削除・編集可能
- **対策**: テーブル別に SELECT/INSERT/UPDATE/DELETE ポリシーを定義

#### 6. **レート制限なし**
- **問題**: API 呼び出しに制限がない
- **影響**: ボット攻撃、リソース枯渇
- **対策**: IP ベースまたはセッションベースの制限実装

#### 7. **データ監査ログ欠落**
- **問題**: 誰がいつ何を変更したかの記録がない
- **影響**: 不正行為の追跡不可、問題発生時の原因特定困難
- **対策**: 監査テーブル（`audit_log`）追加

#### 8. **画像ファイルサイズ制限不足**
- **問題**: Base64 画像が大容量の場合、DBとネットワーク圧迫
- **影響**: ページ読込遅延、ストレージコスト増加
- **対策**: 
  - クライアント側圧縮（750x1000px, <100KB）
  - DB ストレージから Supabase Storage へ移行

## セキュリティ改善ロードマップ

### 第1段階（優先度: 高、難度: 中）

1. **RLS ポリシー厳格化**
   ```sql
   -- 例：furubon テーブル
   CREATE POLICY "anyone_can_read" ON furubon
     FOR SELECT USING (true);
   
   CREATE POLICY "only_poster_can_delete" ON furubon
     FOR DELETE USING (poster = auth.uid());
   ```
   実装時間: 4-6 時間

2. **パスワードハッシュ化（bcrypt）**
   - `completion_password` → bcrypt ハッシュ化
   - 実装時間: 2-3 時間

3. **入力サニタイズ**
   - XSS 対策、SQL インジェクション対策
   - 実装時間: 3-4 時間

### 第2段階（優先度: 中、難度: 高）

4. **簡易認証システム**
   - セッショントークンベース（OAuth 不要）
   - 一端末・一ユーザーの紐付け
   - 実装時間: 2-3 日

5. **バックエンド API**
   - Node.js/Python サーバー
   - API キー秘密化、レート制限実装
   - 実装時間: 3-5 日

### 第3段階（優先度: 中、難度: 低）

6. **監査ログ**
   - 変更記録テーブル追加
   - 実装時間: 1-2 日

7. **画像最適化**
   - クライアント側圧縮、Supabase Storage 移行
   - 実装時間: 2-3 日
